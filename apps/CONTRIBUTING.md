## How to Contribute

There are many ways you can contribute:

### 1\. Reporting a Bug

* First, search our [existing issues](https://www.google.com/search?q=LINK_TO_YOUR_ISSUES) to check if the bug has already been reported.
* If not, open a new issue. Please use our [Bug Report Template](https://www.google.com/search?q=LINK_TO_YOUR_BUG_REPORT_TEMPLATE_IF_AVAILABLE). Describe the bug in as much detail as possible (steps to reproduce, expected/actual behavior, environment information).

### 2\. Suggesting a New Feature

* Before starting work, please discuss your idea in our [discussions](https://www.google.com/search?q=LINK_TO_YOUR_DISCUSSIONS_OR_ISSUES_AREA) or open a [Feature Request Issue](https://www.google.com/search?q=LINK_TO_YOUR_FEATURE_REQUEST_TEMPLATE_IF_AVAILABLE). This allows us to discuss the best approach together.

### 3\. Submitting Code (Pull Request)

1. **Fork** the repository.
2. **Clone** your forked repository to your local system.
    `git clone https://github.com/YOUR-USERNAME/YOUR-REPO.git`
3. **Create a new branch** for your changes:
    `git checkout -b feature/your-feature-name-or-bugfix/description`
4. **Install dependencies:**
    `npm install` (or `yarn install`)
5. **Make your changes.**
6. **Write tests** that cover your changes (if applicable).
7. **Run tests:**
    `npm test`
8. **Format your code** (optional, but recommended):
    `npm run format` (or corresponding script)
9. **Commit your changes** with a meaningful [commit message](https://www.google.com/search?q=%23commit-message-guidelines).
    `git commit -m "feat: Describe your new feature"`
10. **Push** your changes to your fork:
    `git push origin feature/your-feature-name`
11.**Open a Pull Request** in the main repository. Describe your changes in detail and reference any relevant issues.

## Code Guidelines

### Commit Message Guidelines

We use [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/) for our commit messages. Please use the following prefixes:

* `feat`: A new feature
* `fix`: A bug fix
* `docs`: Documentation-only changes
* `style`: Changes that do not affect the meaning of the code (white-space, formatting, missing semicolons, etc.)
* `refactor`: A code change that neither fixes a bug nor adds a feature
* `test`: Adding missing tests or correcting existing tests
* `chore`: Changes to the build process or auxiliary tools and libraries (e.g., dependency updates)

Example: `feat: add user authentication`
